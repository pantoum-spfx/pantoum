What Claude Code Receives

How PANTOUM builds context for AI-powered error fixing

Every time Claude Code is invoked to fix an error, PANTOUM constructs a structured prompt with three components — giving Claude Code the full picture it needs to produce targeted, safe fixes.

Component 1
Build Output
Complete, unfiltered stderr + stdout from the build command — exactly what a developer would see
Component 2
Context Files
Automatically gathered based on error type: package.json, tsconfig, SPFx configs, and error-referenced source files
Component 3
Constraints & Guardrails
Version restrictions, forbidden commands, approach instructions, and SPFx-specific rules

Build Output Capture

PANTOUM captures the complete, unfiltered output from the build command and sends it directly to Claude Code.

Capture Flow

npm run build Execute build in the SPFx solution directory
▼
Build Fails Exit code ≠ 0
▼
stderr Error messages
+
stdout Build log
▼
Combined & Injected Full output sent to Claude Code as-is

What Claude Code Sees

BUILD OUTPUT: ``` Error - [typescript] src/webparts/MyWebPart.ts(42,5): error TS2564: Property 'context' has no initializer and is not definitely assigned in the constructor. Error - [typescript] src/services/DataService.ts(18,3): error TS2532: Object is possibly 'undefined'. Warning - [eslint] src/utils/helpers.ts(7,1): '@typescript-eslint/no-explicit-any' --- STDOUT --- Starting build... Heft build failed with 2 error(s). ```
Why unfiltered?
Claude Code needs the complete picture — surrounding context, warning patterns, and build metadata all contribute to accurate root cause analysis. Filtering would lose critical information.

Context File Selection

PANTOUM automatically gathers relevant files and lists them in the prompt, so Claude Code knows where to look.

Selection Logic

package.json always tsconfig.json if TS errors .yo-rc.json SPFx config config/package-solution.json SPFx config config/serve.json SPFx config config/heft.json if exists config/rig.json if exists src/webparts/MyWebPart.ts from errors src/services/DataService.ts from errors
Always included
Conditional on error type
Extracted from error paths

In the Prompt

AVAILABLE FILES: - package.json - tsconfig.json - .yo-rc.json - config/package-solution.json - config/heft.json - src/webparts/MyWebPart.ts - src/services/DataService.ts

How Files Are Discovered

Source files are extracted from error output using path pattern matching. If the build says src/webparts/MyWebPart.ts(42,5): error TS2564, PANTOUM extracts src/webparts/MyWebPart.ts and includes it in the file list.

Claude Code can also discover additional files at runtime using the Read and Grep tools — the file list is a starting point, not a boundary.

Constraints & Guardrails

Every prompt includes explicit rules and instructions to keep Claude Code safe and focused.

Version Restrictions
Target SPFx version is specified. Claude Code must not modify .yo-rc.json or @microsoft/* package versions — they have already been set correctly by the deterministic patches.
Forbidden Commands
Explicit blocklist: never run heft eject-webpack or any command with "eject". Never delete config files. Never downgrade dependency versions.
Preserve Intent
"This code was working before the upgrade. Fix for compatibility, not broken logic." Don't loosen TypeScript settings. Don't simplify build scripts. Don't create custom webpack configurations.
Approach Instructions
1. Read ALL files mentioned in errors first
2. Plan ALL fixes based on what you read
3. Apply ALL fixes in one batch (Edit/MultiEdit)
4. Run build ONCE after all fixes applied
Structured Prompts
Carefully engineered prompts with explicit constraints maximize consistency across runs.

Migration Prompts

For package migrations, prompts are loaded from template files in src/templates/*.md.

How It Works

1. aiContext in pantoum.patches.yml The template field specifies which template file to load (e.g. pnp-v4-migration).
▼
2. Template file loaded Markdown template from src/templates/*.md with variable substitution. All migration knowledge is self-contained.
▼
3. Prompt assembled Preamble + template content = complete prompt sent to Claude Code.

Template Variables

{{packageName}} → @pnp/sp {{fromVersion}} → 3.15.0 {{toVersion}} → 4.17.0 {{isRemoval}} → false

Template Content

Self-Contained Instructions
All migration knowledge is embedded in the template file: breaking changes, code patterns, verification steps, and edge cases

Built-in Templates

pnp-v4-migration.md — PnP JS → v4
mgt-migration.md — MGT deprecation
react-17-migration.md — React lifecycle
fluent-ui-migration.md — Fluent UI v7 → v8
gulp-to-heft-migration.md — Gulp → Heft
gulpfile-custom-migration.md — Gulp + env injection
scss-declaration-order-migration.md — SCSS fixes
generic-migration.md — Any other package
